Areas de GIT

Working Directory

El area de trabajo corresponde a la carpeta donde inicializamos el repositorio, donde ocasionalmente utilizaremos algun que otro comando, especificamente git init para la creacion del repositorio. Es dicha area nos centraremos en la creacion, edicion y eliminacion de archivos y carpetas.

Staging Area (INDEX)

El area de control de cambios corresponde a un archivo dentro de la carpeta .git/ donde se agregan los archivos y carpetas a los que le haremos seguimiento dentro del proyecto. Se utiliza para realizar las capturas de codigo (SNAPSHOT) que posteriormente deberan ser confirmadas mediante un commit.

Repositorio (Local/Remoto)

El almacen de cambios corresponde a una estructura de subcarpetas dentro de la carpeta .git/ donde se guardan las confirmaciones de cambios en archivos BLOB (Binary Large Object), que se utilizan mediante referencias para navegar dentro del historial de cambios. Estos se pueden gestionar tanto de manera local como remotamente.

Apuntadores de GIT

HEAD

Apuntador movil que indica donde nos encontramos dentro del historial de confirmaciones, se utiliza para navegar dentro del mismo y como referencia al momento de realizar comparaciones con los commits emitidos con anterioridad. Por defecto se ubica en la ultima confirmacion emitida de la rama actual.

BRANCH

Apuntador Dinamico que siempre se corresponde con el ultimo commit de la linea de tiempo en la que nos encontramos trabajando, se utiliza para implementar nuevas caracteristicas, corregir o revisar problemas en el proyecto e incluso realizar diferentes propuestas, sin comprometer la historia principal.

STASH

Apuntador de la zona temporal de cambios que se utiliza principalmente cuando tenemos cambios que no ameritan un commit, y necesitamos movernos en el historial de confirmaciones. Al aplicar el comando se almacenan los cambios una referencia temporal que apunta al Staging Area y al commit donde nos encontramos.

TAG

Apuntador estatico que hace referencia al commit donde se aplico, se utiliza para facilitar la navegacion dentro del historial de confirmaciones y para el versionado del proyecto. El formato suele ser "vn.m.r" donde n es la version principal, m la subversion que indica caracteristicas o compilacion y r corresponde a las revisiones o correcciones de errores dentro del codigo.

Control de Cambios

RESTORE

Este comando permite reestablecer cualquier archivo a como se encontraba en la ultima captura realizada y en caso que no se haya hecho el git add correspondiente utiliza el ultimo commit de referencia. Tambien podemos utilizarlo para quitar capturas del Staging Area (INDEX) si utilizamos el parametro --staged.

RESET

Este comando elimina los commits desde el ultimo hasta la referencia seleccionada y por defecto el parametro --soft viene implicito en el mismo, estos cambios en el historial no afectan a las otras areas. En caso que necesitemos reestablecer por completo el proyecto podemos utilizar el parametro --hard, teniendo en cuenta que sera practico si no existen los cambios en el repositorio remoto.

REVERT

Este comando genera un commit con las operaciones contrarias al commit seleccionado, ideal para cuando los cambios se encuentran en el repositorio remoto y necesitamos deshacer una parte del historial y no todos los commits correspondientes. En caso de conflicto, la operacion se detendra y deberemos resolver el mismo para proceder.

BISECT

Esta herramienta permite el analisis binario de los commits del proyecto, bastante util para detectar commits en los que hayan introducido errores en el codigo. Simplemente debemos indicar cual de las confirmacion contiene dicho error bisect bad y cual es la ultima version funcional bisect good, para que git recorra los commits y consulta si el codigo es funcional.
